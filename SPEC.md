## 0. これは何のアプリか

科学的根拠に基づくAIパーソナルトレーナー「**デジトレ**」。
ユーザーの体の情報から1日の栄養目標を計算し、食べたものの栄養を算出して、
**PFC管理**（Protein=タンパク質・Fat=脂質・Carbohydrate=炭水化物の3栄養素の管理）を支援する。

**設計の核となる方針**：
- 計算は**必ずコードで実行**する（AIの暗算に頼らない＝数値が毎回ブレない）。
- アドバイスや会話は**AIの柔軟さ**を活かす。
- → 計算・記録は「本物のコード」、相談は「AI」が担当する**ハイブリッド構成**。

## 3. 機能①：オンボーディング（初期設定）

初回、または「リセット」と言われたら、以下を**一つずつ**聞き取る。

1. **基本情報**：性別(male/female)、年齢、身長(cm)、現在の体重(kg)
2. **生活活動強度**：1.2（低）〜 1.9（高）
3. **目的（カロリー増減 goal_offset）**：
   - 減量：-250 / -500 / -750
   - 増量：+250 / +500
   - 維持：0
4. **PFC戦略**：①ウェルネス／②ヘルスケア／③ダイエット特化／④筋肥大・競技者

## 4. 機能②：目標値の計算（計算ロジック）

計算の順番：
1. **BMR**（基礎代謝量＝何もしなくても消費するカロリー）を Mifflin-St Jeor 式で算出
2. **TDEE**（総消費カロリー）＝ BMR × 活動強度
3. **目標カロリー** ＝ TDEE ＋ goal_offset
4. 選んだ**戦略**に応じて P・F・C のグラム数を計算

戦略ごとの配分ルール（※P=4kcal/g, F=9kcal/g, C=4kcal/g で換算）：

| 戦略 | P（タンパク質） | F（脂質） | C（炭水化物） |
|------|------|------|------|
| ①ウェルネス | 目標kcal×15% ÷4 | 目標kcal×25% ÷9 | 目標kcal×60% ÷4 |
| ②ヘルスケア | 体重×1.2g | 目標kcal×25% ÷9 | 残りカロリー ÷4 |
| ③ダイエット特化 | 目標kcal×30% ÷4 | 目標kcal×20% ÷9 | 目標kcal×50% ÷4 |
| ④筋肥大・競技者 | 体重×2.0g | 目標kcal×20% ÷9 | 残りカロリー ÷4 |

※「残りカロリー」＝ 目標kcal −（P×4 ＋ F×9）

参考：プロトタイプの計算コード（考え方のたたき台）

> 注：これは最初のたたき台です。実物の targets.py は同じ計算結果のまま、
> ①戦略を if 連打でなく STRATEGIES という「データの表」で持つ（戦略追加が楽）、
> ②性別を検証してエラーにする、③丸めない、という形に進化しています。
> 下のコードにも、その要点（性別検証・丸めない）を反映済み。

```python
def calculate_daily_targets(gender, age, height, weight, activity_level, goal_offset, strategy):
    # 0. 性別を検証（'male'/'female' 以外は黙って進めずエラーで止める）
    if gender not in ("male", "female"):
        raise ValueError(f"未知の性別です: {gender}")

    # 1. BMR（基礎代謝量）を Mifflin-St Jeor 式で計算
    s = 5 if gender == 'male' else -161
    bmr = 10 * weight + 6.25 * height - 5 * age + s

    # 2. TDEE（総消費カロリー）と 目標カロリー
    tdee = bmr * activity_level
    target_kcal = tdee + goal_offset

    # 3. 戦略ごとのPFC配分
    p_g, f_g, c_g = 0, 0, 0
    if strategy == "①ウェルネス":
        p_g = (target_kcal * 0.15) / 4
        f_g = (target_kcal * 0.25) / 9
        c_g = (target_kcal * 0.60) / 4
    elif strategy == "②ヘルスケア":
        p_g = weight * 1.2
        f_g = (target_kcal * 0.25) / 9
        c_g = (target_kcal - (p_g * 4 + f_g * 9)) / 4
    elif strategy == "③ダイエット特化":
        p_g = (target_kcal * 0.30) / 4
        f_g = (target_kcal * 0.20) / 9
        c_g = (target_kcal * 0.50) / 4
    elif strategy == "④筋肥大・競技者":
        p_g = weight * 2.0
        f_g = (target_kcal * 0.20) / 9
        c_g = (target_kcal - (p_g * 4 + f_g * 9)) / 4

    # ★丸めない（正確な数値のまま返す）。丸めるのは表示する側の役目。
    #   丸めた数字を後の計算（摂取量・残り許容量）に使うと誤差が積み重なるため。
    return {
        "BMR": bmr,
        "TDEE": tdee,
        "Target Kcal": target_kcal,
        "P(g)": p_g,
        "F(g)": f_g,
        "C(g)": c_g,
    }
```

## 5. 機能③：摂取量の計算（食べたものの算出）

- ユーザーが食べたもの（食品名・量）を入力する。
- **食品データ**（食品ごとのPFC・カロリー表）から該当食品を引き当てる。
- 量に応じて**合計カロリー・PFCをざっくり算出**する。暗算せずコードで計算。
- これが機能④の「実績」になる。

### 実装状況（2026-06-21）— ファイル: intake.py / foods.csv

- **データ置き場：foods.csv**（食品名, kcal, P, F, C）。値はすべて **100gあたり**。
  - コードと分離しているので、食品を増やすのは CSV に1行足すだけ（コードは触らない）。
  - 中身は八訂をもとにした暫定12品。将来、公式値で精査・拡張する。
- **load_foods(csv_path=foods.csv)**：CSVを読み、「食品名 → {kcal, P, F, C}」の辞書を返す。
  - csv.DictReader で読み、数値は float() に変換。CSVの場所は __file__ 基準で特定
    （どの作業フォルダから実行しても foods.csv を見つけられる）。
- **calculate_intake(items, foods=None)**：items は (食品名, グラム) のリスト。
  - 各食品を `100gあたりの値 × (グラム ÷ 100)` で換算し、kcal/P/F/C を合計して返す。
  - 結果は丸めない（表示側で丸める）。foods 省略時は load_foods() を自動で呼ぶ。
  - 知らない食品名は ValueError で弾く（黙って無視しない）。
- **入力方法について（決定の経緯）**：「カツカレー大盛り」のような“料理ざっくり入力”は
  現時点では未対応。素材＋グラム指定のみ。理由と将来方針は上部「料理でざっくり入力」を参照。
- テスト：test_intake.py（4件）合格。100g一致／合計の手計算一致／未知食品はエラー／空は0。

## 6. 機能④：食事コンサル（夕食限定ではなく食事全体）

- 計算式：**残り許容量 ＝ 目標（②）− 実績（③の合計）**
- 「1日の残り許容量」と「不足タンパク質量(g)」を明示する。
- 朝・昼・夜・間食のどのタイミングでも相談できる。
- コンビニ／外食／自炊の**現実的な選択肢**を提案。
- 不足分は次の食事や翌日で**柔軟に補完**する案を出す。

### 実装状況（2026-06-21）— ファイル: consult.py

- **calculate_remaining(targets, intake)**：目標から実績を引いて「残り」を返す。
  - targets は機能②(calculate_daily_targets)の辞書、intake は機能③(calculate_intake)の辞書。
  - 両者でキー名が違うため、ここで対応づける
    （目標 "Target Kcal"/"P(g)"/"F(g)"/"C(g)" ⇔ 実績 "kcal"/"P"/"F"/"C"）。
  - 戻り値の符号の意味：プラス＝まだ食べてよい量／不足分、マイナス＝超過（食べ過ぎ）。
  - 結果は丸めない（表示側で丸める）。
- **show_remaining(remaining)**：残りを表示。カロリーは「あと○kcal／○kcalオーバー」、
  タンパク質は不足分を強調、脂質・炭水化物は参考として残りを表示。
- **コードとAIの線引き（重要）**：コードは“残りの数値”を出すまでを担当。
  「では夕食はこれ」という**提案文はAIの役目**（将来、ローカルLLM導入時に追加）。
  この分け方により、AIを足しても consult.py の数値部分はそのまま使える。
- テスト：test_consult.py（3件）合格。基本の引き算／食べ過ぎはマイナス／未食なら残り=目標。

## 8. 機能⑥：安全警告

- 月の体重変化が **5%** を超えるような極端な設定には警告を出す。
- 計算の前提：**体重1kgの増減 ≒ 約7200kcal**（体脂肪ベースの一般値）。
- 計算式：
  - 月の予測変化(kg) ＝ 1日のカロリー増減 × 30 ÷ 7200
  - その絶対値 ÷ 現体重 × 100 ＝ 変化率(%)
  - 変化率が 5% を超えたら警告。
- 補足：標準のメニュー（最大 ±750kcal）は通常5%未満に収まる。警告は主に
  「自分で極端な値を入れた／体重が軽い」ケースの歯止めになる。

## 9-2. 機能⑧：可視化

貯めた記録（機能⑤）をグラフにして、変化を一目で振り返れるようにする。
記録がたまって初めて意味が出るので、記録の**次の段階**で作る。

まず作る2つのグラフ：

1. **カロリー実績 vs 目標（棒グラフ）**
   日付ごとに「実際に食べたカロリー」と「目標カロリー」を並べ、
   食べすぎ／不足が一目でわかるようにする。
2. **タンパク質(P)の達成度（折れ線または棒グラフ）**
   日付ごとの「実績P ÷ 目標P」を追い、不足しがちなタンパク質を継続管理する。

実現方法（どちらにするかは段階⑥で決定）：
- **簡単な形**：記録CSVを Excel／スプレッドシートで開いてグラフ化（コードを増やさない）。
- **アプリ組み込み**：Python のグラフ作成ライブラリ（matplotlib など）でアプリ自身が描画。
